# Backend Changelog

## [Unreleased]

### Added

#### Search & discovery (Sprint 3 — `epic/search-and-discover`)
- `rooms` table gains a `category` column — `ENUM('sports','study','entertainment','social')`, nullable, default `NULL` (`database/init/02-rooms.sql`).
- `POST /api/v1/rooms` and `PATCH /api/v1/rooms/:room_id` accept an optional `category`. Values outside the enum return `400 VALIDATION_ERROR`; `PATCH` accepts `null` to clear it.
- `GET /api/v1/rooms/hall` now supports discovery query params:
  - `category` — filter by one of the enum values (`400` on an unknown value).
  - `q` — case-insensitive keyword match against room title / description.
  - `sort_by` — one of `created_at` (default), `event_time`, `member_count` (`400` on an unknown value); `order` — `asc` / `desc`.
  - Cursor semantics depend on `sort_by`: an ISO timestamp for `created_at`, an opaque numeric offset for the other sorts.
- `formatHallRoom` / `formatMyRoom` / `formatRoomDetails` now return the `category` field.
- `GET /api/v1/rooms` accepts an optional `role` filter — `owner` (rooms the caller created) or `member` (rooms the caller joined but does not own); `400` on any other value. Backed by the pure `roleFilterClause()` helper (unit-tested).

#### Room status & lifecycle (Sprint 4 — `epic/room-status`)
- `rooms.status` is `ENUM('open','recruiting_closed','cancelled')`. A frontend-facing display status is derived live by `computeDisplayStatus()` — priority `cancelled` > `ended` (by `event_end_time`) > `in_progress` (by `event_time`) > `recruiting_closed` > `open`. "Ended" / "in_progress" are computed from event times, not stored.
- `POST /api/v1/rooms/:room_id/recruiting/close` / `.../recruiting/open` (owner only) — pause and resume recruiting. While closed, `POST /:room_id/join` fails with `RECRUITING_CLOSED`; existing members and all owner actions are unaffected. `recruiting_closed` rooms are hidden from `GET /hall` but still listed in `GET /` ("my rooms").
- Capacity is **not** a status: `ROOM_FULL` (join) and the `is_full` hall flag are derived live from `member_count >= max_members`. A room at capacity can still be `open` or `recruiting_closed`.

#### Waitlist (manual promotion)
- `room_members.approval_status` gains a `waitlisted` value (`database/init/03-room_members.sql`).
- `POST /:room_id/join` on a full room no longer returns `ROOM_FULL` — the user is added with `approval_status = 'waitlisted'` and the response carries `status: "waitlisted"`. Waitlisted members do not count toward capacity (`member_count` / `is_full` still count `approved` only). Status precedence: full → `waitlisted`, else approval-required → `pending`, else `approved` (see `computeJoinStatus()`, unit-tested).
- `GET /:room_id/waitlist` — owner lists waitlisted users, ordered by `joined_at` (FIFO).
- `POST /:room_id/waitlist/:user_id/promote` — owner promotes a waitlisted user to `approved`; returns `400 ROOM_FULL` if no spot is free, `404 WAITLIST_ENTRY_NOT_FOUND` if the user is not waitlisted.
- `DELETE /:room_id/members/:user_id` (owner kick) now matches any membership row (approved / pending / waitlisted), so owners can also clear waitlist entries.

#### Join-request / approval flow
- `GET /api/v1/rooms/:room_id/requests` — list pending join requests (owner only).
- `POST /:room_id/requests/:user_id/approve`, `POST /:room_id/requests/approve-all`, `POST /:room_id/requests/:user_id/reject` — owner manages pending requests.
- `DELETE /:room_id/members/:user_id` — owner kicks an approved member.
- `room_members.approval_status` (`pending` / `approved`) drives the flow; rooms with `join_approval_required = true` enqueue join requests instead of admitting immediately.

#### Web push notifications
- `database/init/06-push_subscriptions.sql` — `push_subscriptions` table.
- `GET /api/v1/push/vapid-public-key`, `POST /push/subscribe`, `GET /push/subscriptions`, `POST /push/test` (`src/routes/v1/push.ts`), backed by `src/lib/push-notifications.ts` (`web-push`, VAPID keys from `VAPID_PUBLIC_KEY` / `VAPID_PRIVATE_KEY`).

### Changed
- **Auth is opaque-token based, not JWT.** Tokens are 64-char hex strings generated by `src/lib/token.ts` and validated by lookup in `user_tokens`; logout sets `user_tokens.revoked_at`. There is no `src/lib/jwt.ts` / `jose` dependency. (Supersedes the JWT description in the original entry below.)
- Auth middleware lives at `src/routes/v1/middleware/auth.ts`; on every request it rejects revoked tokens and updates `last_used_at` / `user_agent` / `ip_address`.
- `GET /api/v1/auth/me` added — returns the authenticated user.
- `DELETE /api/v1/rooms/:room_id` (dismiss, owner only) sets `status = 'cancelled'` (the original entry said `'closed'`, which is not a valid enum value).

## [0.1.0] — 2026-04-17 (initial backend)

### Added

#### Infrastructure
- `src/index.ts` — Hono app entry point; mounts CORS, request logger, and both API routers; serves on `PORT` env var (default `5050`)
- `src/db/connection.ts` — mysql2 connection pool (`DB_HOST`, `DB_PORT`, `DB_DATABASE`, `DB_USER`, `DB_PASSWORD`)
- `src/db/types.ts` — TypeScript row interfaces: `UserRow`, `RoomRow`, `RoomMemberRow`, `UserTokenRow`
- `src/lib/errors.ts` — `apiError()` helper; all error responses follow `{ error: { code, message, details } }`

> **Note:** the original release notes described JWT auth (`src/lib/jwt.ts`, `jose`). The implementation shipped with opaque tokens instead — see the **Changed** section above.

#### Auth routes (`/api/v1/auth`)
- `POST /register` — validates all fields, checks duplicate email/username/phone, hashes password with `Bun.password.hash`, issues an access token
- `POST /login` — identifier (email, username, or phone) + password login; issues an access token
- `POST /logout` — revokes the current token by setting `user_tokens.revoked_at`
- `POST /passkey/register/start` — stub (501)
- `POST /passkey/register/finish` — stub (501)
- `POST /passkey/login/start` — stub (501)
- `POST /passkey/login/finish` — stub (501)
- `DELETE /passkey/:passkey_id` — stub (501)
- `GET /passkey/list` — stub (501)

> Passkey endpoints require `@teamhanko/passkeys-sdk` or `@simplewebauthn/server` to be implemented.

#### Room routes (`/api/v1/rooms`)
- `GET /` — list rooms the authenticated user has joined; cursor-based pagination
- `GET /hall` — public room listing; supports `include_joined` and `include_full` query flags; cursor-based pagination
- `POST /` — create a room; owner is automatically inserted as the first member
- `POST /:room_id/join` — join a room; enforces capacity and duplicate-membership checks
- `POST /:room_id/leave` — leave a room; owner is blocked (must dismiss instead)
- `DELETE /:room_id` — dismiss a room (owner only)

#### Database schema
- `database/init/01-users.sql` — creates `users`, `rooms`, `room_members`, `user_tokens` tables with FK constraints and cascade deletes
